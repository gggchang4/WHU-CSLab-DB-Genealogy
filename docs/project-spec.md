## 2. 项目背景

### 2.1 项目名称

`"寻根溯源" 族谱管理系统`

### 2.2 课程背景

本项目为武汉大学计算机学院数据库课程实践项目，主题为应用设计实践，重点考察：

- 数据库需求分析
- 概念设计与逻辑设计
- 关系模式规范化
- SQL 编写能力
- 大规模数据生成能力
- 物理优化与索引设计
- 简单应用界面开发与演示能力

## 3. 业务场景描述

系统用于管理多个族谱。每个族谱对应一个家族，族谱需要记录家族成员信息、血缘关系与婚姻关系，并支持多用户协作维护。

### 3.1 族谱基本信息

每个族谱至少包含以下基本信息：

- 谱名
- 姓氏
- 修谱时间
- 创建用户

### 3.2 多用户协作

系统支持多用户：

- 每个用户都可以创建自己的族谱
- 族谱创建者可以邀请其他用户共同参与维护
- 被邀请用户可以为对应族谱增加成员或参与编辑

### 3.3 家族成员信息

族谱中的成员至少需要记录以下基本信息：

- 姓名
- 性别
- 生卒年
- 生平简介

### 3.4 关系信息

系统需要记录以下关系：

- 血缘关系
  - 父子
  - 父女
  - 母子
  - 母女
- 婚姻关系

## 4. 核心挑战

PDF 中明确指出本题的核心挑战包括：

1. 成员之间的层级关系深度不确定
2. 存在同名同姓但不同辈分的情况
3. 需要快速查询某个成员的所有直系后代或追溯祖先
4. 需要快速查询两个人名之间是否存在亲缘关系，以及两者之间的亲缘链路

对开发的直接启示：

- 不能仅依赖成员姓名作为唯一识别条件
- 需要为成员设置稳定唯一标识，例如 `member_id`
- 关系查询需要支持递归或图遍历能力
- 数据规模较大时，需要关注索引、查询路径和执行计划

## 5. 功能需求

## 5.1 简单应用界面

系统需要实现一个基础图形化界面。PDF 允许的技术路线包括但不限于：

- Python
  - Flask
  - Django
- .NET
  - WinForm
  - WPF
  - WebAPI
- 其他开发工具

说明：课程要求并未强制限定具体框架，但必须能够演示要求中的功能。

## 5.2 用户登录

系统需要支持用户登录，并满足以下权限要求：

- 用户只能看到自己创建的族谱
- 用户只能看到自己被邀请参与的族谱
- 用户不能操作与自己无关的族谱

## 5.3 用户管理

用户采用注册制，至少需要支持：

- 用户注册
- 用户登录
- 用户身份识别

## 5.4 Dashboard

系统需要提供仪表盘，至少展示：

- 家族总人数
- 男女比例

说明：这里的“家族”可以理解为当前查看族谱对应的家族。

## 5.5 族谱管理

系统需要支持族谱的增删改查，即：

- 创建族谱
- 查看族谱
- 编辑族谱
- 删除族谱

同时需要支持：

- 族谱邀请他人参与创作

## 5.6 成员管理

系统需要支持成员的增删改查，即：

- 新增成员
- 查看成员
- 编辑成员
- 删除成员

成员查找需要支持：

- 按人名模糊查找

## 5.7 树形预览

系统需要支持以层级结构或缩进列表展示某个分支的人物关系树状图。

最低要求：

- 能从某个成员或某个分支节点出发
- 展示该分支的层级关系
- 展示结果具有明确父子代际结构

## 5.8 人物祖先查询

系统需要支持输入一个人物的 `ID`，并以树状图展示其父辈以上的祖先。

最低要求：

- 输入条件为成员唯一标识，不应只使用姓名
- 输出结果为向上追溯的祖先树

## 5.9 人物亲缘关系查询

系统需要支持输入两个人的 `ID`，并查询两者之间是否存在亲缘关系通路。

如果存在，应展示：

- 是否存在亲缘关系
- 两者之间的亲缘链路

说明：这里更接近“关系路径查询”，本质上是图路径问题或递归查询问题。

## 6. 数据库建模与规范化设计要求

## 6.1 ER 图绘制

需要识别实体及属性，并绘制 ER 图。

PDF 中给出的实体示例包括：

- 成员
- 家庭

ER 图中需要标明联系类型，例如：

- `1:1`
- `1:N`
- `M:N`

说明：在本题实际实现中，除了“成员”“族谱/家族”“用户”之外，通常还会涉及“邀请”“婚姻关系”“亲子关系”等对象或联系。

## 6.2 关系模式转换

需要将 ER 图转换为关系表，并说明其达到的范式级别。

课程建议：

- 至少满足 `3NF`
- 或达到 `BCNF`

## 6.3 约束设计

需要定义：

- 主键
- 外键
- `CHECK` 约束

PDF 中给出的约束示例：

- 父亲的出生年份必须早于子女

建议在实现中进一步考虑的约束包括：

- 母亲出生年份早于子女
- 成员不能与自己建立父母关系
- 婚姻关系两端不能为同一成员
- 受邀协作者必须是系统中已注册用户

## 7. 数据生成与导入导出要求

## 7.1 模拟数据生成

需要编写 Python 脚本或 SQL 存储过程生成模拟数据，并满足以下规模要求：

- 至少生成 `10` 个族谱
- 至少有 `1` 个族谱拥有 `50,000` 以上成员
- 整个系统中不少于 `100,000` 条模拟成员数据

同时满足以下结构要求：

- 每一个族谱内的成员至少与另一个成员有亲缘关系
- 单个族谱内至少模拟出 `30` 代传承关系

这意味着：

- 数据不能是纯随机孤立成员
- 必须有真实的关系链结构
- 至少一个大族谱需要足以支撑性能测试与递归查询验证

## 7.2 导入导出

需要演示 PostgreSQL 的 `COPY` 能力，包括：

- 将 CSV 数据批量导入数据库
- 导出某个分支的备份文件

说明：PDF 明确提到了 PostgreSQL 的 `COPY`，因此最终实现和实验材料中最好围绕 PostgreSQL 展开。

## 8. SQL 核心功能要求

需要编写满足以下需求的 SQL 语句，并且每个需求要求使用单个 SQL 实现。

## 8.1 基本查询

给定一个成员 `ID`，查询：

- 其配偶
- 其所有子女

## 8.2 递归查询

需要使用递归公用表表达式 `Recursive CTE`。

需求：

- 输入成员 `A` 的 `ID`
- 输出其向上追溯的所有历代祖先

这是本项目的核心 SQL 能力之一。

## 8.3 统计分析

需要完成以下统计类 SQL：

1. 统计某个家族中平均寿命最长的一代人
2. 查询所有年龄超过 `50` 岁且没有配偶的男性成员
   - 年龄可假设由当前日期减出生日期得到
3. 找出家族中出生年份早于该辈分平均出生年份的所有成员

说明：

- “某辈分”要求系统中存在可用于分代统计的逻辑依据
- 若设计中不直接存储“代数”，则需要通过关系递归推导

## 9. 物理优化与索引设计要求

## 9.1 索引策略

需要针对以下场景设计合适索引：

- 根据姓名模糊查询
- 根据父节点 `ID` 查询子节点

## 9.2 性能对比

需要记录在“有索引”和“无索引”两种情况下，执行以下查询的时间差异：

- 查询某曾祖父的所有曾孙
- 即四代查询

并提交：

- 执行时间对比
- `EXPLAIN` 执行计划分析

## 13. 推荐的数据对象清单

说明：本节是为了便于后续建模，不是 PDF 原文逐字要求。

建议至少考虑以下核心对象：

- `users`
  - 系统用户
- `genealogies`
  - 族谱
- `genealogy_collaborators`
  - 族谱协作者或邀请关系
- `members`
  - 家族成员
- `parent_child_relations`
  - 亲子关系
- `marriages`
  - 婚姻关系

建议关注的字段示例：

- 用户：用户名、密码哈希、显示名、创建时间
- 族谱：谱名、姓氏、修谱时间、创建者、创建时间
- 成员：姓名、性别、出生年、去世年、生平简介、所属族谱
- 亲子关系：父母成员 ID、子女成员 ID、关系类型
- 婚姻关系：成员 A、成员 B、婚姻状态或备注

## 14. 面向大模型的开发约束摘要

如果后续使用大模型协助编码、建模或写 SQL，可以直接遵守以下约束：

- 所有成员必须使用稳定唯一 ID 进行引用
- 不得将姓名作为唯一主键或唯一业务标识
- 所有查询必须考虑同名同姓的情况
- 祖先查询应优先使用递归 CTE 实现
- 亲缘链路查询应支持路径展示，而不只是布尔判断
- 成员数据规模需要支持 `100,000+`
- 至少一个族谱规模需要达到 `50,000+`
- 单个族谱需支持 `30` 代以上关系深度
- 数据生成不能生成孤立成员
- 姓名模糊检索和父节点查子节点必须有索引策略
- 需要保留实验报告、SQL 截图、执行计划和导出文件等验收材料

## 15. 当前阶段的默认实现假设

以下假设是为了减少后续实现分歧，后续可按项目进展调整：

- 默认 RDBMS 采用 PostgreSQL
- 默认使用 Web 形式的基础图形界面，而不是桌面端
- 默认所有复杂关系查询以成员 `ID` 为输入
- 默认“族谱”是权限和数据隔离的核心边界
- 默认 Dashboard 先围绕单个族谱展示统计数据
- 默认 README 与开发文档会随着项目迭代持续更新

## 16. 一句话总结

这个课程项目本质上是一个面向大规模家谱关系数据的多用户协作式族谱管理系统，需要同时完成应用开发、数据库设计、数据工程、复杂 SQL、性能优化和实验验收材料准备。
